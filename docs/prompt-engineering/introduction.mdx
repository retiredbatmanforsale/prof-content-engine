# Prompt engineering — Introduction

Learn how to **talk to models** so they help you reliably—not just once in a while.

---

## What you will learn

| Focus | In plain terms |
|-------|----------------|
| **Prompt engineering** | Shaping instructions and context so the model does the *right* job |
| **Working with AI** | Treating the system as a partner you **steer**, not magic you **hope** for |
| **A simple workflow** | Breaking big asks into **small steps** you can review and improve |

---

## Thinking with generative AI

- **You own the goal.** The model suggests text; you judge quality, facts, and fit.
- **Be specific.** Vague asks get vague answers; constraints and examples narrow the space.
- **Iterate.** First output is a draft; refine with follow-up prompts or edits.

---

## From big problem to finished work

Break work into stages. Each stage gets its **own prompt** (or short chat turn).

| Stage | You ask the model to… | Example prompt (short) |
|:-----:|----------------------|-------------------------|
| **Idea** | Explore options, angles, or outlines | “List 5 headline angles for a blog on remote onboarding.” |
| **Refine** | Pick direction, add constraints | “Keep only B2B SaaS; rank by novelty.” |
| **Draft** | Produce a first version | “Write a 400-word draft using angle 2; friendly tone.” |
| **Iterate** | Fix gaps, tone, or structure | “Shorten intro; add one bullet on security.” |
| **Produce** | Final format (email, slide bullets, checklist) | “Turn this into 5 speaker notes + 1 takeaway line each.” |

---

## Examples: vague vs usable

| Too loose | Better |
|-----------|--------|
| “Write about AI.” | “Explain prompt engineering to **non-technical managers** in **under 200 words**; use **3 bullets**.” |
| “Make it better.” | “Tighten to **120 words**, **active voice**, **no jargon**; keep the same three main ideas.” |
| “Code something.” | “**Python** function that reads a CSV path and returns **row count**; include **type hints** and a **one-line docstring**.” |

---

## Takeaway

**Prompt engineering** is less about clever phrases and more about **clear goals, structure, and iteration**—the same habits that make human collaboration work.
