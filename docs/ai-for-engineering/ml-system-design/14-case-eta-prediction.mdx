## 1. The prompt

> **"Design ETA prediction for Uber."**

A regression problem. Given a request — pickup point, drop-off point, current time — predict how long the ride will take. The output is a number of minutes shown to the rider in the UI before they confirm the booking. This case is structurally different from recsys (no ranking), RAG (no generation), and fraud (no classification): it is point prediction over a deeply spatial-temporal input space.

---

## 2. The wrinkles — what's different from the flagships and other case studies

### 2.1 Geographic features are central

Every primitive of the model is geographic. The input is two points and a time. Useful features include road-graph features (distance along the optimal route, number of turns, number of intersections), spatial-bucket features (H3 hexagons, geohash), and historical aggregates (mean ETA for this origin-destination pair at this time of day). **Treating lat / long as plain numerical features is L4 ceiling** — the model cannot generalise across cities and route shapes without explicit spatial inductive bias.

### 2.2 Temporal-spatial features

Real-time conditions dominate the residual after the static features explain the bulk: current traffic state, weather, road incidents, recent events (a concert just ended in this hexagon). These features are computed in **streaming aggregators with millisecond-to-minute freshness** — the feature store from Lesson 5, configured for spatial-temporal aggregations.

### 2.3 Asymmetric loss

Under-prediction is worse than over-prediction. If the displayed ETA is 8 minutes and the actual is 14, the rider is angry, the driver looks bad, and the trip earns a 1-star rating. If the displayed is 14 and the actual is 11, the rider is mildly happy. **MSE penalises both directions equally** — which is the wrong product behaviour. The fix: **pinball loss** at an asymmetric quantile, or a custom asymmetric loss function.

### 2.4 Quantile outputs, not point estimates

The production UI shows two numbers: "typically 12 minutes" (P50) and "up to 18 minutes if traffic" (P90). A single point estimate cannot communicate uncertainty; the rider has no way to plan. **Quantile regression with explicit P50 and P90 outputs** is the L5 default. Using a single point regression and showing it as "12 ± something" is L4.

## 3. Walking the framework

### 3.1 Define the problem

Metric tree:

- **Business goal** — rider satisfaction (rating, repeat rate), driver utilisation (low idle time), trip completion rate
- **Product metric** — mean absolute error of displayed-vs-actual ETA, P90 over-shoot rate (how often actual exceeds the displayed P90), rider complaint rate, cancellation rate post-booking
- **ML objective** — pinball-loss-trained quantile regression at $\tau = 0.5$ as primary (the displayed point estimate), $\tau = 0.9$ as secondary (the displayed buffer); calibration of the quantiles against held-out actuals as a guardrail

The unit is **a single ETA prediction** for one (origin, destination, time, conditions) input. There is no funnel — the input fan-in is bounded by the request, and the output is a small number of quantile values.

### 3.2 Data pipeline

Inherits the feature store + temporal-split discipline from Lesson 2. ETA-specific:

- **H3 spatial bucketing** — both pickup and drop-off encoded as H3 hex IDs at multiple resolutions (city-scale, neighbourhood-scale, block-scale)
- **Road-graph features** — distance along the optimal route, number of turns, intersection count, historical congestion at each hex along the route
- **Streaming temporal-spatial aggregations** — current avg speed in each hex over the last 5 minutes; weather state per hex; active incident reports
- **Spatial cross-validation** — never split randomly. Either hold out entire cities (test generalisation across cities) or hold out time-blocks within cities (test temporal generalisation). Random split allows leakage because nearby trips on the same day are highly correlated.

### 3.3 Model architecture

A practical production stack is a **gradient-boosted tree** (XGBoost / LightGBM) on engineered spatial features for the primary score, optionally augmented with a small neural network on graph embeddings (H3 hex embeddings, road-segment embeddings) for tail cases. Because the dominant features are already-engineered streaming aggregations, GBDTs perform competitively with deep models — same observation as Lesson 13.

For quantile outputs, **train two models** (one for P50, one for P90) or use a **single multi-quantile model** with separate output heads per quantile. The L5 candidate names both options; production teams typically use multi-head for parameter sharing.

### 3.4 Train and evaluate

- **Loss design**: pinball loss at quantile $\tau$:

  $$
  \rho_\tau(y - \hat{y}) = \max\big(\tau \cdot (y - \hat{y}),\ (\tau - 1) \cdot (y - \hat{y})\big)
  $$

  At $\tau = 0.5$, pinball reduces to mean absolute error (symmetric). At $\tau = 0.9$, it penalises under-predictions 9× more than over-predictions, encouraging the model to over-shoot in that head.

- **Eval**: MAE on the P50 head against actuals; **calibration of the P90** measured by the empirical hit rate of the actual being below the predicted P90 (should be ~90%); per-segment metrics (city, time of day, distance bucket); spatial holdout MAE.

- **Spatial cross-validation** is not optional. A model that performs well on a random split can fail catastrophically on a held-out city because it has memorised the local spatial patterns it saw in training.

### 3.5 Deploy and serve

Inherits the latency budget from Lesson 5. ETA-specific:

- **Sub-100ms p99** typical, since the prediction blocks the booking confirmation flow
- **Streaming feature store** for the temporal-spatial features (avg speed per hex, weather, incidents) — refreshed every minute or faster
- **Caching on origin-destination-time tuples** for repeated common queries (popular routes during rush hour repeat thousands of times per minute)
- **Graceful degradation** to a distance-based heuristic if the model or the feature store is unavailable

### 3.6 Monitor and maintain

Inherits Lesson 7. ETA-specific:

- **Online learning / fast retraining** for road incidents — when a major road closure or accident is reported, the model needs updated features within minutes, not hours. The feature store handles most of this; the model itself is retrained daily or hourly on the latest residuals.
- **Per-city drift monitoring** — different cities drift independently (a new highway opens in Bangalore, traffic patterns shift in São Paulo); monitor per-city PSI and trigger city-level retraining
- **P90 calibration drift** as a first-class signal — if the P90 hit rate drops below 85% (target 90%), the calibration is stale and the buffer is no longer protecting users from surprise ETA over-shoots

## 4. Back-of-envelope math

### 4.1 Pinball loss worked example

For a single observation $y = 12$ minutes, prediction $\hat{y} = 10$ minutes, at $\tau = 0.9$:

$$
y - \hat{y} = 2 \text{ (positive — under-prediction)}
$$

$$
\rho_{0.9}(2) = \max(0.9 \times 2,\ -0.1 \times 2) = \max(1.8,\ -0.2) = 1.8
$$

For $\hat{y} = 14$ (over-prediction by 2):

$$
\rho_{0.9}(-2) = \max(0.9 \times (-2),\ -0.1 \times (-2)) = \max(-1.8,\ 0.2) = 0.2
$$

The same magnitude of error costs **9× more** when the prediction is too low than too high. This is what pushes the P90 head to over-shoot — exactly the asymmetric behaviour the product needs.

### 4.2 H3 hex area scaling

H3 resolution 8 hexes are roughly 0.74 km² in area (about a city block); resolution 9 are ~0.10 km²; resolution 10 are ~0.015 km² (intersection-scale). For pickup-point bucketing in dense urban areas, resolution 9 or 10 is typical; for cross-city aggregations, resolution 7 or 6. **The L5 candidate names a specific resolution** based on the use case — generic "use H3" is L4.

### 4.3 Spatial cross-validation cost

Random splits routinely show 2-5× lower MAE than spatial holdouts because of leakage from nearby same-day trips. Reporting random-split MAE as the model's quality is **a real production failure mode** — teams have shipped models that looked good in eval and failed catastrophically on the first new city. The L5 candidate flags this without being asked.

## 5. What the interviewer is actually grading (case-specific)

- **Did you propose quantile regression unprompted?** Single-point regression with MSE is L4 ceiling.
- **Did you name pinball loss?** Naming the loss makes the asymmetric handling concrete.
- **Did you mention P50 / P90 outputs to the UI?** Communicating uncertainty to the rider is a product decision, not just a metric.
- **Did you mention H3 or geohash spatial bucketing?** Treating lat / long as plain features is L4.
- **Did you propose spatial cross-validation?** Random splits are the canonical leakage failure for spatial models.
- **Did you stream real-time traffic / weather features?** Static features cannot capture road incidents.

Failure modes to avoid:

- **MSE on a single point estimate.** Wrong loss + wrong output shape = double L4 ceiling.
- **Random train/test split.** Leakage from spatially correlated trips inflates eval scores.
- **Lat / long as raw features.** No spatial generalisation possible without an explicit spatial inductive bias.
- **No real-time features.** Static models cannot react to road incidents.
- **No P90 calibration monitoring.** A miscalibrated P90 silently fails users.
