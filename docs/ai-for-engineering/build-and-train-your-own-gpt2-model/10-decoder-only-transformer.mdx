## The logit lens — peek inside the stack

Now that you've seen how a prompt moves through the decoder, here's the natural follow-up question: **does the model "know" the answer at layer 6, or does it only commit at layer 12?**

The logit lens (Nostalgebraist, 2020) is a one-line trick that answers this. After the forward pass, instead of decoding only the final layer through the LM head, decode *every* intermediate layer's hidden state through the same head. Each cell below is the model's top guess for that position *if you had stopped the forward pass right there*.

Try it on `The capital of France is`:

- **Bottom layers** (closer to the embedding) usually emit token-shaped noise — copies of the input or near-neighbours in embedding space.
- **Mid layers** start to surface plausible candidates. You can sometimes watch the right answer crystallize.
- **Top layer** matches the model's actual prediction (the brightest cell — high certainty).

Saturation = top-1 probability. Hover any cell for the top-5 candidates with their probabilities.

## Step 1 — `model.py`: the full GPT

The architecture in pseudocode:

```text
GPT(idx):
    x = TokenEmbedding(idx) + PositionalEmbedding(0..T)   # (B, T, n_embd)
    x = Dropout(x)
    for block in n_layer Blocks:
        x = Block(x)                                       # pre-LN attn + MLP, both with residuals
    x = LayerNorm(x)                                       # final LN
    logits = LMHead(x)                                     # (B, T, vocab_size), tied to TokenEmbedding
    return logits
```

Three architectural details that Lesson 9 didn't cover but matter for assembly:

### Final LayerNorm

After the last `Block`, we apply one more LayerNorm before the output projection. Reason: the residual stream's magnitude has grown across 12 blocks. LN re-establishes unit variance one last time so the LM head sees normalised inputs. Adds $2C = 1{,}536$ parameters. Standard in every modern decoder-only LM.

### Weight tying

The token embedding matrix has shape $(V, C) = (50257, 768)$. The LM head is a `Linear(C, V)` whose weight has shape $(V, C)$ — *the same shape*. **Tie them**: set `lm_head.weight = embedding.weight`. One matrix used twice — input embedding and output unembedding share parameters. Saves ~38M parameters and empirically gives slightly better validation perplexity ([Press & Wolf, 2017](https://arxiv.org/abs/1608.05859)).

### Initialisation that matches GPT-2

Two pieces:

- All `Linear` weights and `Embedding` weights initialised to $\mathcal{N}(0, 0.02)$, biases (where they exist) zeroed.
- The output projection of attention (`attn.proj`) and MLP (`mlp.proj`) — the things that *write into* the residual stream — get a smaller std: $0.02 / \sqrt{2 L}$. Without this, the residual stream's variance grows with depth (each block adds, variance accumulates), and you'd need extra warmup to compensate.

```python
# nanogpt-from-scratch/model.py

import torch
import torch.nn as nn
import torch.nn.functional as F

from blocks import Block
from config import GPTConfig
from embeddings import Embeddings


class GPT(nn.Module):
    def __init__(self, config: GPTConfig):
        super().__init__()
        self.config = config

        self.embed = Embeddings(config)
        self.drop = nn.Dropout(config.dropout)
        self.blocks = nn.ModuleList([Block(config) for _ in range(config.n_layer)])
        self.ln_f = nn.LayerNorm(config.n_embd)
        self.lm_head = nn.Linear(config.n_embd, config.vocab_size, bias=False)

        # Tie the unembedding to the token embedding.
        self.lm_head.weight = self.embed.tok.weight.weight

        # Init.
        self.apply(self._init_weights)
        for name, p in self.named_parameters():
            if name.endswith("attn.proj.weight") or name.endswith("mlp.proj.weight"):
                torch.nn.init.normal_(p, mean=0.0, std=0.02 / (2 * config.n_layer) ** 0.5)

    def _init_weights(self, m):
        if isinstance(m, nn.Linear):
            nn.init.normal_(m.weight, mean=0.0, std=0.02)
            if m.bias is not None:
                nn.init.zeros_(m.bias)
        elif isinstance(m, nn.Embedding):
            nn.init.normal_(m.weight, mean=0.0, std=0.02)

    def forward(self, idx, targets=None):
        B, T = idx.shape
        assert T <= self.config.block_size

        x = self.drop(self.embed(idx))
        for block in self.blocks:
            x = block(x)
        x = self.ln_f(x)
        logits = self.lm_head(x)

        loss = None
        if targets is not None:
            loss = F.cross_entropy(
                logits.view(-1, logits.size(-1)),
                targets.view(-1),
            )
        return logits, loss
```

Verification — at the default `GPTConfig()` (GPT-2 small dimensions), the model should report:

| Component        | Params (target)     |
|------------------|---------------------|
| Embeddings       | 39.4M               |
| 12 × Block       | 84.97M              |
| Final LN         | 1,536               |
| LM head (tied)   | 0 (shares with embed) |
| **Total**        | **~124.4M**         |

If you get $124{,}{ }412{,}160$ exactly, you've matched GPT-2 small.

## Step 2 — `train.py`: the training loop

Five sub-problems wrapped in one file: data loading, batching, the forward-backward step, the LR schedule, and evaluation.

### The naive solution — constant LR, no warmup

Use `torch.optim.Adam(lr=3e-4)`, run forever. What goes wrong:

- **Early instability.** At step 1 the model's weights are random; the loss landscape near init is rough. Hitting it with a full learning rate often produces a huge gradient that overshoots into a worse region. Loss spikes, sometimes never recovers.
- **Late stagnation.** A constant LR is too coarse for the late-training regime. The model needs smaller and smaller updates as it converges; a constant `3e-4` overshoots fine-tuning details.

### The motivated solution — AdamW + warmup + cosine decay

Two components:

**AdamW** ([Loshchilov & Hutter, 2017](https://arxiv.org/abs/1711.05101)) is Adam with decoupled weight decay. Standard for modern LLM training — GPT-2, GPT-3, Llama all use it. Decay coefficient typically $0.1$; betas $(0.9, 0.95)$ rather than the defaults $(0.9, 0.999)$ because language models converge fast and the second moment shouldn't have so much memory.

**Linear warmup, then cosine decay**:

$$
\eta(t) = \begin{cases}
\eta_{\max} \cdot \dfrac{t}{T_{\text{warm}}} & t < T_{\text{warm}} \\[6pt]
\eta_{\min} + \tfrac{1}{2}(\eta_{\max} - \eta_{\min})\!\left(1 + \cos\!\left(\pi \cdot \dfrac{t - T_{\text{warm}}}{T - T_{\text{warm}}}\right)\right) & T_{\text{warm}} \leq t \leq T
\end{cases}
$$

Warmup gives the model a few hundred steps of small updates while the parameters are still randomly initialised — they get a chance to find a stable region of the loss landscape before the optimiser unleashes full power. Cosine decay then lowers the LR smoothly toward $\eta_{\min}$ over the remaining training.

### The training-loop code

```python
# nanogpt-from-scratch/train.py  (excerpts — full file in scaffold)

import math, os, time, urllib.request
import torch

from config import GPTConfig
from generate import generate
from model import GPT
from tokenizer import decode, encode

# ---- Hyperparameters (Mac-laptop friendly: smaller than GPT-2 small) ----
config = GPTConfig(
    n_embd=192, n_head=6, n_layer=6, block_size=128, dropout=0.0,
)
batch_size, max_iters, eval_interval = 32, 3000, 500
warmup_iters, lr_max, lr_min = 100, 3e-4, 3e-5
weight_decay, beta1, beta2, grad_clip = 0.1, 0.9, 0.95, 1.0
device = ("mps" if torch.backends.mps.is_available()
          else "cuda" if torch.cuda.is_available() else "cpu")

# ---- Data: download + tokenize TinyShakespeare ----
DATA = "data/tinyshakespeare.txt"
if not os.path.exists(DATA):
    os.makedirs(os.path.dirname(DATA), exist_ok=True)
    urllib.request.urlretrieve(
        "https://raw.githubusercontent.com/karpathy/char-rnn/master/data/tinyshakespeare/input.txt",
        DATA,
    )
data = torch.tensor(encode(open(DATA).read()), dtype=torch.long)
n = int(0.9 * len(data))
train_data, val_data = data[:n], data[n:]

def get_batch(split):
    src = train_data if split == "train" else val_data
    ix = torch.randint(len(src) - config.block_size - 1, (batch_size,))
    x = torch.stack([src[i : i + config.block_size] for i in ix])
    y = torch.stack([src[i + 1 : i + 1 + config.block_size] for i in ix])
    return x.to(device), y.to(device)

# ---- Model ----
model = GPT(config).to(device)
optimizer = torch.optim.AdamW(model.parameters(), lr=lr_max,
                              betas=(beta1, beta2), weight_decay=weight_decay)

def lr_at(step):
    if step < warmup_iters:
        return lr_max * step / warmup_iters
    if step >= max_iters:
        return lr_min
    progress = (step - warmup_iters) / (max_iters - warmup_iters)
    return lr_min + 0.5 * (lr_max - lr_min) * (1.0 + math.cos(math.pi * progress))

# ---- Train ----
t0 = time.time()
for step in range(max_iters + 1):
    for g in optimizer.param_groups:
        g["lr"] = lr_at(step)

    if step % eval_interval == 0:
        # ... estimate train/val loss, sample 200 tokens, print ...
        pass

    x, y = get_batch("train")
    _, loss = model(x, y)
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    torch.nn.utils.clip_grad_norm_(model.parameters(), grad_clip)
    optimizer.step()

torch.save(model.state_dict(), "checkpoint.pt")
```

The full file (with the eval and sampling-every-500-steps logic) is in the scaffold under `train.py`.

A few things to flag:

- **We shrank the model.** GPT-2 small is 124M parameters; that's too big to train in 10 minutes on a laptop. We use `n_embd=192, n_head=6, n_layer=6, block_size=128` — about 11M parameters. Same architecture, fewer of every dimension. Loss curve looks the same; absolute loss is higher because the model is smaller.
- **`zero_grad(set_to_none=True)`.** Faster than zeroing tensors. Standard in modern PyTorch.
- **`clip_grad_norm_`.** Caps the gradient's L2 norm at 1.0. Prevents the occasional gradient explosion that can otherwise wreck training in 10 steps.
- **No DataLoader, no num_workers.** TinyShakespeare is 1MB; we keep it in RAM and slice random windows. For larger datasets you'd want a streaming pipeline.

## Math by typing it

### Cross-entropy loss

For a batch of $B \times T$ predictions over a $V$-sized vocabulary, with logits $z \in \mathbb{R}^{B \times T \times V}$ and targets $y \in \mathbb{Z}^{B \times T}$:

$$
\mathcal{L} \;=\; -\frac{1}{BT} \sum_{b, t} \log \frac{\exp(z_{b, t, y_{b, t}})}{\sum_{v} \exp(z_{b, t, v})}
$$

Equivalently: average negative log-probability of the correct next token, over all positions. For a random model on a 50,257-token vocab, $\mathcal{L} \approx \log V \approx 10.83$ — that's where loss starts. Anything below is real learning.

### AdamW update rule

For each parameter $\theta$ with gradient $g$:

$$
m \leftarrow \beta_1 m + (1 - \beta_1) g, \qquad v \leftarrow \beta_2 v + (1 - \beta_2) g^2
$$

$$
\hat m = \frac{m}{1 - \beta_1^t}, \qquad \hat v = \frac{v}{1 - \beta_2^t}
$$

$$
\theta \leftarrow \theta \;-\; \eta \!\left( \frac{\hat m}{\sqrt{\hat v} + \epsilon} \;+\; \lambda \theta \right)
$$

The decoupled weight decay $\lambda \theta$ is added to the update *separately* from the gradient — it doesn't get scaled by $\sqrt{\hat v}$. That's the AdamW vs vanilla Adam difference, and it matters at scale.

### Cosine LR schedule

After warmup ($t \geq T_{\text{warm}}$):

$$
\eta(t) \;=\; \eta_{\min} + \tfrac{1}{2}(\eta_{\max} - \eta_{\min})\!\left(1 + \cos\!\left(\pi \cdot \frac{t - T_{\text{warm}}}{T - T_{\text{warm}}}\right)\right)
$$

Smooth from $\eta_{\max}$ at $t = T_{\text{warm}}$ down to $\eta_{\min}$ at $t = T$. The cosine is differentiable at both endpoints — no LR-restart shock. Used by GPT-2, GPT-3, Llama, and most modern LLM training runs.

## Compute receipts

Training run on a 2024 MacBook (M3 Pro, 36 GB):

| Quantity                    | Value                                  |
|-----------------------------|----------------------------------------|
| Architecture                | GPT-2 shape, scaled-down               |
| Parameters                  | 11.1M (vs GPT-2 small's 124M)          |
| Training data               | TinyShakespeare, 338K tokens           |
| Iterations                  | 3,000                                  |
| Batch size × block_size     | 32 × 128 = 4,096 tokens/step           |
| Tokens seen total           | ~12.3M (≈36 epochs of TinyShakespeare) |
| Training time (MPS)         | ~9 minutes                             |
| Training time (CPU)         | ~25 minutes                            |
| Final train loss            | ~1.47 nats                             |
| Final val loss              | ~1.57 nats                             |
| Perplexity (val)            | $e^{1.57} \approx 4.8$                 |
| Peak GPU memory (MPS)       | ~700 MB                                |
| Checkpoint size             | ~45 MB                                 |

The val/train gap (1.57 vs 1.47) is small because we're using GPT-2's BPE tokenizer rather than a TinyShakespeare-specific one — the 50K vocab is mostly under-utilised, so the model has room to memorise without overfitting per-token.

For comparison, GPT-2 small trained on WebText reaches val loss ~3.05 (perplexity ~21) — but on a vocabulary that covers all of English, not just Shakespeare. You can't directly compare; the loss numbers only make sense within a fixed (data, vocab, tokenization) trio.
