Lesson 4's executor loop looked like this:

```python
for sq in sub_questions:
    findings[sq] = self._react(sq)   # executor 2 waits for executor 1
```

The sub-questions are independent — none of them need the results of any other. There is no reason executor 2 should wait for executor 1. With five sub-questions and two search steps each, sequential execution takes 10 time units. With parallel execution it takes 2 — the depth of one executor, run simultaneously with all the others.

This lesson rewrites the executor loop with `asyncio.gather()`.

## asyncio in 60 seconds

`asyncio` is Python's built-in concurrency model for I/O-bound work. It runs on a single thread but switches between tasks when one is waiting (for a network call, a database read, a search API). That's exactly the pattern for LLM and search calls.

```python
import asyncio

# A coroutine — defined with async def, called with await
async def fetch(url):
    result = await some_async_http_call(url)
    return result

# Run multiple coroutines concurrently
results = await asyncio.gather(fetch(url1), fetch(url2), fetch(url3))
# All three run at the same time; results is a list in the same order
```

Two rules:
- Functions that use `await` must be declared with `async def`
- `asyncio.gather(*coroutines)` runs all of them concurrently and waits for all to finish

In this lesson, `_react()` becomes `async def _react()`, `run()` becomes `async def run()`, and the `for` loop becomes `asyncio.gather()`.

## The timing model

```
Lesson 4 (sequential):
  Executor 1 ████████
  Executor 2         ████████
  Executor 3                 ████████
  Executor 4                         ████████
  Executor 5                                 ████████
  Total wall time:  5 × 2 = 10 units

Lesson 5 (parallel):
  Executor 1 ████████
  Executor 2 ████████
  Executor 3 ████████
  Executor 4 ████████
  Executor 5 ████████
  Total wall time:  2 units  (max depth of one executor)
```

Total compute is the same — 10 units of work. Wall-clock time collapses.

## Now look at what isolation costs

The parallel executors run faster. But they run in complete isolation — each starts with an empty messages list, no knowledge of what the others are retrieving.

## What just happened

Both executors searched for the same AlphaCode documents. Neither knew the other had already done so. They ran in parallel, which is fast — but they also both paid the cost of the same retrieval.

For five executors with overlapping sub-topics, this gets worse. And it is not just about efficiency: if two executors retrieve slightly different versions of the same fact (different documents, different phrasings), the synthesiser gets contradictory inputs.

A shared memory layer changes this: executors check before searching ("has anyone already retrieved AlphaCode docs?"), post their findings as they go, and the synthesiser reads from one consistent source of truth instead of N independent caches.
