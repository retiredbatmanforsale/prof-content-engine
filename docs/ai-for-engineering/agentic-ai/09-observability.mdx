Lesson 8 ended with a loop that ran correctly and produced no insight. Three retries, three FAILs — and you could not tell which executor was responsible, how many LLM calls were made, or whether the third attempt was any different from the second.

This is the standard failure mode of multi-agent systems: they work (or fail) as a black box. Adding a **Tracer** changes that. Every LLM call, every search, every retry, every verdict gets logged as a `TraceEvent`. At the end of a run, `tracer.summary()` prints a table that shows you exactly where the system spent its effort.

## Instrumentation, not rewriting

The Tracer does not replace any existing component. It is additive: you pass a `tracer` instance into the Orchestrator, which passes it down to `_react` and `Critic`. At each key point you add one line:

```python
self.tracer.log(agent_id, "llm_call")
self.tracer.log(agent_id, "search", {"query": sq})
self.tracer.log("critic", "critic_verdict", {"passed": verdict.passed})
```

The components themselves don't change. The Tracer is a passive observer — it records, it doesn't act.

This is the standard pattern for production observability: instrument at boundaries (every external call, every decision point) without modifying the core logic.

## TraceEvent — the unit of observation

```python
@dataclass
class TraceEvent:
    agent_id: str            # who logged this: "exec_1_2", "orchestrator", "critic"
    event:    str            # what happened: "llm_call", "search", "plan", "critic_verdict"
    payload:  object = None  # optional metadata — a dict or None
    ts:       float  = field(default_factory=time.time)  # wall-clock timestamp
```

Four fields. `agent_id` and `event` are the grouping keys for the summary. `payload` carries structured metadata (e.g. `{"passed": True, "n_gaps": 0}`) that can be inspected per-event. `ts` lets you compute latency between events when needed.

The `field(default_factory=time.time)` pattern defers the `time.time()` call to event creation time. If you wrote `ts: float = time.time()` instead, every `TraceEvent` would share the same timestamp — the one captured when the class was defined.

---

## What to instrument

Five points in the system capture almost all meaningful signal:

| Where | Event | What it tells you |
|-------|-------|-------------------|
| `Orchestrator.run()` — after `planner.plan()` | `"plan"` | How many sub-questions the planner produced |
| `Orchestrator.run()` — top of each attempt loop | `"attempt_start"` | How many retries the system made |
| `Orchestrator.run()` — before `_synthesise()` | `"synthesise"` | How many LLM calls the synthesiser used |
| `Orchestrator._react()` — before each `llm.chat()` | `"llm_call"` | Per-executor LLM usage |
| `Orchestrator._react()` — when a `SEARCH:` fires | `"search"` | Per-executor search usage |
| `Critic.review()` — after `CriticVerdict` is created | `"critic_verdict"` | Pass/fail history across retries |

The agent ID for executors encodes both attempt number and position: `f"exec_{attempt+1}_{i}"`. This lets you distinguish `exec_1_2` (attempt 1, sub-question 2) from `exec_2_0` (retry attempt, gap sub-question) in the summary table.

---

## What tracer.summary() should print

After a successful two-attempt run on the AlphaCode/Codex query, the summary should look like this:

```
────────────────────────────────────────────────────
agent                event              count
────────────────────────────────────────────────────
critic               critic_verdict         2
exec_1_0             llm_call               2
exec_1_0             search                 1
exec_1_1             llm_call               2
exec_1_1             search                 1
exec_1_2             llm_call               6
exec_1_2             search                 5
exec_2_0             llm_call               2
exec_2_0             search                 1
orchestrator         attempt_start          2
orchestrator         plan                   1
orchestrator         synthesise             2
────────────────────────────────────────────────────
Total LLM calls: 17   Total searches: 8   Attempts: 2
```

`exec_1_2` used 6 LLM calls and 5 searches. The other executors used 2 each. That is the bottleneck: the impact sub-question sent the executor into a multi-step search loop that never resolved on attempt 1. `exec_2_0` (the retry) resolved in 2 steps because the search index now had impact data.

Without the tracer, you see two lines of verbose output and a final PASS. With it, you know *why* attempt 1 failed and *where* to look if it keeps failing.

## Reading the trace

After implementing the TODOs, `exec_1_2` should stand out immediately: six LLM calls and five searches against two each for `exec_1_0` and `exec_1_1`. That is the system telling you something: sub-question 3 (research impact) sent the executor into a multi-step search loop that exhausted every reformulation before giving up.

`exec_2_0` shows the retry worked in two steps. The gap was not caused by a bad sub-question — it was caused by a missing search index on attempt 1. The retry hit the `"impact"` key and resolved immediately.

Without the trace, both of these facts are invisible. With it, you know:
- **Where to look** — executor 2 on attempt 1
- **Why it failed** — not reasoning failure, but index miss
- **That the fix worked** — exec_2_0 cost 2 calls, not 6

This is what observability gives you: hypotheses become testable.

## From mock to real: one line

Every component in the system calls `self.llm.chat(messages)` through the `LLMClient` interface. Swapping the mock for a real model is a one-line change in `main()`:

```python
# Mock (what you used for all nine lessons):
llm = MockLLM()

# Real — Anthropic Claude:
import anthropic
class AnthropicLLM(LLMClient):
    def __init__(self, model="claude-opus-4-7"):
        self.client = anthropic.Anthropic()
        self.model  = model
    def chat(self, messages):
        resp = self.client.messages.create(
            model=self.model, max_tokens=1024, messages=messages[1:],
            system=messages[0]["content"],
        )
        return resp.content[0].text

llm = AnthropicLLM()
```

The `SearchTool` interface works the same way: swap `MockSearchEngine` for a real web search API, a vector database, or a document retrieval system. The agent code does not change.
