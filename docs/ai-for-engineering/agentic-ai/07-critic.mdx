Lesson 6 ended with a synthesiser that invented a confident claim — AlphaCode's approach influencing StarCoder and CodeLlama — from a sub-question that returned no data at all. Shared memory kept retrieval consistent. It said nothing about whether the answer was trustworthy.

A **Critic** is an agent whose only job is to read the synthesised answer and audit it against the actual findings. It does not search, does not reason step by step, does not rewrite anything. It answers one question: *does this answer hold up?*

## The verdict format

The Critic outputs structured text that downstream code can parse. Three line types:

```
GAP: <sub-question that produced no useful data>
UNSUPPORTED: <claim in the answer not backed by any finding>
VERDICT: PASS
VERDICT: FAIL
```

One `VERDICT:` line always appears last. `GAP:` and `UNSUPPORTED:` lines may appear zero or more times before it. A `VERDICT: FAIL` means at least one gap or unsupported claim was found.

Structured output is a design choice — not a feature of LLMs. The system prompt instructs the Critic to use exactly these prefixes so the `CriticVerdict` class can parse them with simple string operations. No regex, no JSON schema required.

---

## The Critic prompt

```python
CRITIC_PROMPT = """You are a research critic. You receive:
  1. The original research question
  2. The findings for each sub-question (what was actually retrieved)
  3. The synthesised answer to review

Audit the synthesised answer for two types of problems:

GAP: a sub-question whose finding contained no useful data, but the answer
     treats as if the gap does not exist or invents information to fill it.

UNSUPPORTED: a specific claim in the synthesised answer that does not appear
             in any of the provided findings. Quote the claim exactly.

Output one line per issue found (using the GAP: or UNSUPPORTED: prefix).
Then output exactly one of:
  VERDICT: PASS
  VERDICT: FAIL

VERDICT: FAIL if any GAP or UNSUPPORTED issue was found. PASS otherwise."""
```

## CriticVerdict — parsing structured output

```python
class CriticVerdict:
    def __init__(self, raw: str):
        self.raw         = raw
        self.passed      = "VERDICT: PASS" in raw
        self.gaps        = [l[4:].strip()  for l in raw.splitlines() if l.startswith("GAP:")]
        self.unsupported = [l[12:].strip() for l in raw.splitlines() if l.startswith("UNSUPPORTED:")]
```

Three fields derived from the raw text: `passed` (bool), `gaps` (list of strings), `unsupported` (list of strings). Any code downstream — like a retry loop in Lesson 8 — can read `verdict.passed` without touching the raw string.

## What the Critic does. What it can't do.

The Critic has one job: audit. It reads, it reasons, it flags. It does not search, it does not rewrite, it does not retry. Its output is a structured verdict that *other code* is supposed to act on.

<div className="not-prose my-6 rounded-xl border-l-4 border-rose-500 bg-rose-50/70 py-3 pl-4 pr-3 dark:bg-rose-950/25">
  <p className="m-0 text-sm font-semibold text-rose-900 dark:text-rose-200">Why structured output matters here</p>
  <p className="mt-2 mb-0 text-sm text-slate-800 dark:text-slate-200">
    <code>verdict.passed</code> is a bool. <code>verdict.gaps</code> is a list. The orchestrator in Lesson 8 will check <code>if not verdict.passed</code>, iterate over <code>verdict.gaps</code> to re-search specific sub-questions, and pass <code>verdict.unsupported</code> to a revised synthesis call. None of this is possible if the Critic returns freeform prose — you cannot branch code on a paragraph.
  </p>
</div>

This is a general principle for multi-agent systems: **agents that produce outputs consumed by other agents must produce structured outputs.** Freeform prose is for humans. Parseable formats are for systems.
