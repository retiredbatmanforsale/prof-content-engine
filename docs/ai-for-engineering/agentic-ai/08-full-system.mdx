Lesson 7 ended with a `VERDICT: FAIL` that went nowhere. The Critic found a gap and an unsupported claim, printed them clearly — and then the agent delivered the bad answer unchanged.

The Critic can only diagnose. Something else has to act. The **Orchestrator** is that something: a control-flow layer that sits above the whole pipeline and owns the retry loop. When the Critic says FAIL, the Orchestrator re-searches the flagged gaps, feeds the new findings back into synthesis, and re-runs the Critic. It stops when the verdict passes, or when `max_retries` runs out.

## The feedback loop

```
Query
  │
  ▼
Planner ──► sub_questions
                 │
          ┌──────▼────────────────────────────────┐
          │  Executors (parallel) + SharedMemory  │
          └──────┬────────────────────────────────┘
                 │ findings (accumulated)
                 ▼
           Synthesiser ──► answer
                 │
                 ▼
            Critic ──► CriticVerdict
                 │
        ┌────────┴───────────────────┐
 passed │                             │ failed
        ▼                             ▼
     return                   sub_questions = verdict.gaps
                               attempt += 1  →  back to Executors
                               (or return after max_retries)
```

Three decisions per loop iteration:

1. **What to execute** — on retry, only the gap sub-questions, not all of them
2. **What to synthesise from** — all findings accumulated across all attempts (new results fill the gaps without discarding what already worked on attempt 1)
3. **When to stop** — `verdict.passed` is True, `verdict.gaps` is empty, or `max_retries` exhausted

---

## Why re-search only the gaps?

On attempt 1, two sub-questions were answered correctly. Sub-question 1 (AlphaCode's benchmark) retrieved solid data. Sub-question 2 (algorithmic comparison) retrieved solid data. Only sub-question 3 (research impact) came back empty.

Re-running all three sub-questions on attempt 2 would re-fetch data that is already correct. The gap is specific. The Critic told us exactly which sub-question failed. `verdict.gaps` is a list — use it.

The Orchestrator keeps a running `all_findings` dict. New results from the retry are merged in. The synthesiser on attempt 2 sees sub-questions 1, 2, and 3 — the first two from the original pass, the third freshly retrieved. A strictly better picture than attempt 1.

---

## MockLLM and MockSearchEngine get one update

The `MockSearchEngine` gains an `"impact"` entry. When the gap sub-question is re-searched with a query containing `"impact"`, it now returns real data instead of `"No documents found"`.

The synthesiser (`_synth`) checks whether `"copilot"` or `"1000"` appears in the findings. If it does, it uses the retrieved impact data instead of inventing the StarCoder/CodeLlama claim. The Critic on attempt 2 finds no gap and no unsupported claim — `VERDICT: PASS`.

This makes the main playground deterministic: attempt 1 fails, attempt 2 passes.

    def __str__(self):
        status = "✓ PASS" if self.passed else "✗ FAIL"
        lines  = [f"Critic: {status}"]

    async def get(self, key: str):

## Now look at what the loop can't tell you

The Orchestrator on the happy path (attempt 1 fails, attempt 2 passes) gives you a correct answer and a `VERDICT: PASS`. That's good. But what happens on a query where the gap *never* resolves — where the search engine genuinely has no data, and three attempts all return the same empty result?

## The system is complete. You can't debug it.

Every component exists and works in isolation. The Planner decomposes queries. The Executors retrieve and reason in parallel. SharedMemory deduplicates. The Synthesiser assembles. The Critic audits. The Orchestrator retries. The pipeline runs end to end.

What you have no visibility into:

- **Per-agent timing** — which executor takes 3× as long as the others on every run
- **LLM call count** — how many calls the system made in total (hint: more than you think)
- **Retry diagnostics** — whether gap sub-questions were re-searched with different queries or identical ones
- **Verdict progression** — whether each retry attempt improved (fewer gaps, different unsupported claims) or was identical to the last

Without this, tuning the system is guesswork. You can't know whether `max_retries=2` is too low or wasteful, whether the Planner is generating resolvable sub-questions, or where to focus optimization effort.

Observability is not optional in multi-agent systems. It's the instrumentation layer that makes everything else debuggable. That's Lesson 9.
