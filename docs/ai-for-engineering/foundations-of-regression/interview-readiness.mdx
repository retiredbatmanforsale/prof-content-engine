## Linear Regression

<InterviewQuestion
  number={1}
  difficulty="Conceptual"
  company={["Google", "Amazon", "Meta", "Microsoft"]}
  question="What are the assumptions of linear regression? How would you test each one?"
>

The six assumptions and how to verify them:

1. **Linearity** вАФ The relationship between X and E[Y|X] is linear. Test with residual-vs-fitted plots; a curved pattern indicates non-linearity. Add polynomial features or transform Y if needed.

2. **Independence of errors** вАФ Residuals are uncorrelated. Test with the Durbin-Watson statistic (value вЙИ 2 means no autocorrelation; values below 1.5 or above 2.5 are a red flag). Critical for time-series data.

3. **Homoscedasticity** вАФ Variance of residuals is constant across fitted values. Test with a residual-vs-fitted plot (a "cone" shape indicates heteroscedasticity) or the Breusch-Pagan test. Fix with log/Box-Cox transform on Y, or use Weighted Least Squares.

4. **Normality of residuals** вАФ Errors are normally distributed. Test with a QQ-plot of residuals. Note: this assumption is only needed for valid t-tests and CIs on small samples вАФ OLS is unbiased regardless.

5. **No perfect multicollinearity** вАФ Features are not exact linear combinations of each other. Test with the Variance Inflation Factor (VIF); VIF above 5 warrants investigation, VIF above 10 is serious. Fix with Ridge regression or dropping the highest-VIF feature.

6. **Exogeneity** вАФ E[ќµ|X] = 0; errors are uncorrelated with the predictors. Violated by omitted variable bias or measurement error. Hard to test directly вАФ requires domain knowledge.

**Google interviewers push hardest on why each assumption matters. Amazon asks what happens when each is violated.** A complete answer chains through the assumption, the diagnostic tool, and the fix.

</InterviewQuestion>

<InterviewQuestion
  number={2}
  difficulty="Math"
  company={["Amazon", "Google"]}
  question="Derive the closed-form OLS solution. Under what conditions does it fail, and how do you fix it?"
>

**Derivation:**

Minimise the sum of squared residuals: `SSE = вАЦy вИТ Xќ≤вАЦ¬≤`

Taking `вИВSSE/вИВќ≤ = 0`:

`вИТ2XбµА(y вИТ Xќ≤) = 0` вЖТ `XбµАXќ≤ = XбµАy`

Solving: **`ќ≤ћВ = (XбµАX)вБї¬єXбµАy`** вАФ the normal equation.

This is also the MLE under Gaussian noise: assuming `ќµ ~ N(0, ѕГ¬≤I)`, maximising the log-likelihood gives the same normal equations.

**When does it fail?**

`XбµАX` is not invertible when:
- **p &gt; n** (more features than samples) вАФ the matrix is rank-deficient
- **Perfect multicollinearity** вАФ one feature is a linear combination of others

**Fixes:**
- **Ridge regression**: `ќ≤ћВ_ridge = (XбµАX + ќїI)вБї¬єXбµАy` вАФ adding `ќїI` makes the matrix invertible regardless of rank
- **Lasso / ElasticNet**: use coordinate descent instead of a closed form
- **PCA + regression**: project features to a lower-dimensional orthogonal basis first
- **`np.linalg.lstsq`**: uses SVD вАФ numerically preferred over explicit matrix inversion even when `XбµАX` is invertible

</InterviewQuestion>

<InterviewQuestion
  number={3}
  difficulty="Math"
  company="Amazon"
  question="Can you fit a linear regression model with 100 rows and 1,000 features? What happens to the OLS solution?"
>

This is a confirmed Amazon Applied Scientist phone-screen question (Bengaluru, May 2025).

**The answer: no unique OLS solution exists.**

When p &gt; n (1,000 features, 100 rows), `XбµАX` is a 1,000 √Ч 1,000 matrix of rank at most 100. It is rank-deficient and therefore **not invertible** вАФ the normal equation `XбµАXќ≤ = XбµАy` has infinitely many solutions.

Practically: the model perfectly fits the training data with zero training error but generalises catastrophically вАФ pure overfitting.

**Fixes:**

| Method | Mechanism | When to use |
|---|---|---|
| **Ridge (L2)** | `ќ≤ћВ = (XбµАX + ќїI)вБї¬єXбµАy` вАФ `ќїI` guarantees invertibility | Correlated features, keep all |
| **Lasso (L1)** | Coordinate descent with soft-thresholding | Suspect sparse signal, want feature selection |
| **ElasticNet** | Combines L1 + L2 | Both sparsity and multicollinearity |
| **PCA + OLS** | Project to k &lt;&lt; p orthogonal components | Features highly correlated |
| **Bayesian regression** | Place a prior on ќ≤ вАФ effectively regularises | Need uncertainty quantification |

The interviewer typically follows up: *"Why does adding ќїI to XбµАX fix invertibility?"* вАФ because a positive definite matrix (ќїI) plus any positive semi-definite matrix (XбµАX) is always positive definite, hence invertible.

</InterviewQuestion>

<InterviewQuestion
  number={4}
  difficulty="Math"
  company={["Google", "Amazon", "Meta", "Microsoft"]}
  question="Why does L1 regularization produce sparsity while L2 does not? Give a geometric explanation."
>

This is the single most-attributed regression question across all four companies вАФ confirmed at Google MLE (2026), Meta MLE (verbatim on Prepfully), Microsoft MLE (Prepfully), and standard at Amazon.

**Geometric explanation:**

Regularized regression minimises Loss(ќ≤) subject to a constraint on ќ≤. The constraint defines a feasible region:

- **L2 (Ridge)**: constraint is `вАЦќ≤вАЦвВВ¬≤ вЙ§ t` вАФ a **circle** (2D) or sphere (nD). Smooth, no corners.
- **L1 (Lasso)**: constraint is `вАЦќ≤вАЦвВБ вЙ§ t` вАФ a **diamond** (2D) or cross-polytope (nD). Has corners exactly on the coordinate axes.

The optimum is where the loss contours first touch the constraint region. The L1 diamond's corners sit on the axes вАФ where one or more `ќ≤_j = 0`. Loss contours (ellipses) are far more likely to first touch a corner than a smooth edge. This is why **Lasso solutions are exactly sparse**: coefficients hit zero, not just shrink toward it.

**Analytical explanation:**

The L1 subgradient at `ќ≤_j = 0` is the interval `[вИТќї, ќї]`. The KKT condition allows a coefficient to stay at exactly zero if its gradient falls within this interval. L2 has a smooth gradient (`2ќїќ≤_j`), so weights shrink toward but never reach zero for finite ќї.

**Practical implications:**

- L1 вЖТ feature selection (sparse ќ≤); L2 вЖТ all features kept with shrinkage
- With correlated features: Ridge shrinks both together; Lasso picks one arbitrarily
- **Bayesian view**: L2 corresponds to a Gaussian prior; L1 corresponds to a Laplace prior вАФ the sharp peak at zero induces sparsity

</InterviewQuestion>

<InterviewQuestion
  number={5}
  difficulty="Conceptual"
  company={["Google", "Amazon", "Meta"]}
  question="How do you detect multicollinearity? What is VIF, what threshold matters, and how do you fix it?"
>

Confirmed at Meta, Amazon, and Google.

**Detection tools:**

1. **Correlation matrix** вАФ pairwise correlations above 0.8 are suspicious, but doesn't catch multivariate collinearity
2. **Condition number of XбµАX** вАФ eigenvalue ratio; above 30 suggests moderate, above 100 suggests severe multicollinearity
3. **Variance Inflation Factor (VIF)** вАФ the standard tool:

`VIF_j = 1 / (1 вИТ R¬≤_j)`

where `R¬≤_j` is the R¬≤ from regressing feature j on all other features. VIF above 5 warrants investigation; above 10 is serious.

**A classic Amazon variant:** *"Given features: year-born, BMI, country-at-birth, weekly-alcohol, nationality-at-birth вАФ what's the problem?"* The trap is spotting near-perfect collinearity between country-at-birth and nationality-at-birth without being told.

**Fixes:**

- Drop the highest-VIF feature iteratively and recheck
- Apply **Ridge regression** вАФ the standard fix; spreads weight across correlated features stably
- Combine correlated features via PCA into orthogonal components

**Important nuance (Interview Query):** *"How would you interpret a negative coefficient for a variable that's positively correlated with Y?"* This is Simpson's paradox вАФ a coefficient reflects the *partial* effect holding other features constant. Multicollinearity and confounders can flip signs.

</InterviewQuestion>

## Logistic Regression

<InterviewQuestion
  number={6}
  difficulty="Conceptual"
  company={["Google", "Microsoft"]}
  question="Explain logistic regression. Why is it called 'regression' if it's a classifier? Derive the sigmoid from the logit."
>

Confirmed Google 2026 MLE question (LinkJob report) and Microsoft DS intern Glassdoor question.

**Why "regression":**

Logistic regression models the **log-odds (logit)** of class membership as a *linear* function of features. The regression is over continuous log-odds вАФ the name reflects that linear relationship, not the output type.

**Deriving the sigmoid:**

Start with the logit: `log(p / (1вИТp)) = wбµАx + b`

Solve for p step by step:

`p = e^(wбµАx + b) ¬Ј (1 вИТ p)`

`p ¬Ј (1 + e^(wбµАx + b)) = e^(wбµАx + b)`

`p = 1 / (1 + e^(вИТ(wбµАx + b))) = ѕГ(wбµАx + b)`

The sigmoid emerges directly from the logit model вАФ it's not arbitrary. It maps вДЭ вЖТ (0,1) smoothly, is differentiable everywhere, and has the elegant derivative `ѕГ'(z) = ѕГ(z)(1вИТѕГ(z))` вАФ critical for backpropagation.

**Logistic regression is a discriminative model** вАФ it directly models P(Y|X), not P(X,Y). Google Research interviewers sometimes follow up citing the Ng & Jordan 2001 result that LR asymptotically outperforms Naive Bayes as data grows.

</InterviewQuestion>

<InterviewQuestion
  number={7}
  difficulty="Math"
  company={["Google", "Amazon", "Meta"]}
  question="Why don't we use MSE for logistic regression? Give both the non-convexity argument and the vanishing gradient argument."
>

Confirmed Google MLE follow-up (LinkJob 2026), standard at Amazon, and Analytics Vidhya / ProjectPro. **A strong answer requires both arguments вАФ one alone is incomplete.**

**Argument 1 вАФ Non-convexity:**

Let `≈Ј = ѕГ(wбµАx)`. The MSE cost is `J = ќ£(y вИТ ѕГ(wбµАx))¬≤`. The Hessian involves terms like `ѕГ(1вИТѕГ)(1вИТ2ѕГ(wбµАx))` вАФ when ѕГ saturates near 0 or 1, this can become indefinite. The loss surface has **saddle points and local minima**. Gradient descent can get stuck.

Binary cross-entropy has Hessian `H = XбµАWX` where `W = diag(ѕГбµҐ(1вИТѕГбµҐ))`. Since `ѕГбµҐ(1вИТѕГбµҐ) > 0` for all finite z, H is positive semi-definite вАФ **the loss is convex** and gradient descent finds the global minimum.

**Argument 2 вАФ Vanishing gradients:**

The MSE gradient for a misclassified example with extreme confidence is:

`вИВJ/вИВw вИЭ (y вИТ ≈Ј) ¬Ј ѕГ(z)(1вИТѕГ(z)) ¬Ј x`

When `ѕГ(z) вЖТ 0`, `ѕГ(1вИТѕГ) вЖТ 0` as well. The gradient **vanishes on confidently wrong predictions** вАФ exactly where you need the largest update.

Binary cross-entropy's gradient simplifies to `(≈Ј вИТ y) ¬Ј x` вАФ no `ѕГ'` term. It doesn't saturate and keeps pushing hard on wrong predictions.

</InterviewQuestion>

<InterviewQuestion
  number={8}
  difficulty="Math"
  company={["Meta", "Google"]}
  question="Derive binary cross-entropy from scratch. Show how it follows from MLE under a Bernoulli model."
>

Confirmed at Meta MLE phone screens (Bharathi Priyaa's Medium) and Google MLE (LinkJob 2026).

**Step 1 вАФ Bernoulli likelihood:**

For a single example with label y вИИ `{0, 1}` and predicted probability `p = ѕГ(wбµАx)`:

`P(y | x; w) = p^y ¬Ј (1вИТp)^(1вИТy)`

When y=1 it gives p, when y=0 it gives (1вИТp).

**Step 2 вАФ Joint likelihood over m examples:**

`L(w) = ќ† p_i^y_i ¬Ј (1вИТp_i)^(1вИТy_i)`

**Step 3 вАФ Log-likelihood:**

`log L(w) = ќ£ [ y_i ¬Ј log(p_i) + (1вИТy_i) ¬Ј log(1вИТp_i) ]`

**Step 4 вАФ Negative log-likelihood (MLE вЖТ minimisation):**

`J(w) = вИТ(1/m) ¬Ј ќ£ [ y_i ¬Ј log(p_i) + (1вИТy_i) ¬Ј log(1вИТp_i) ]`

This is binary cross-entropy. **Minimising BCE is exactly maximising the Bernoulli log-likelihood.**

**Step 5 вАФ Gradient (the elegant result):**

`вИВJ/вИВw = (1/m) ¬Ј XбµА(ѕГ(Xw) вИТ y)`

The gradient is simply the prediction error `(≈Ј вИТ y)`, because the sigmoid's derivative cancels with the chain-rule factor вАФ making the gradient of BCE + sigmoid as clean as MSE + identity.

</InterviewQuestion>

<InterviewQuestion
  number={9}
  difficulty="Conceptual"
  company={["Google", "Amazon"]}
  question="What is the Bayesian interpretation of L1 and L2 regularization? How does this connect to MAP estimation?"
>

A Google Research and Amazon Applied Scientist favorite.

**The setup:**

MAP estimation maximises `P(w | data) вИЭ P(data | w) ¬Ј P(w)`. Taking the log:

`log P(w | data) = log-likelihood + log P(w) + const`

The choice of prior `P(w)` determines the regularizer.

**L2 вЖФ Gaussian prior:**

If `P(w) = N(0, ѕД¬≤I)`, then `log P(w) = вИТ(1/2ѕД¬≤) ¬Ј вАЦwвАЦ¬≤`. Adding this to the log-likelihood and negating gives BCE + `ќївАЦwвАЦ¬≤` вАФ **Ridge regularization**. The Gaussian has no sharp spike at zero, so weights shrink but never hit zero.

**L1 вЖФ Laplace prior:**

If `P(w) = Laplace(0, b)`, then `log P(w) = вИТ(1/b) ¬Ј вАЦwвАЦвВБ`. This gives BCE + `ќївАЦwвАЦвВБ` вАФ **Lasso regularization**. The Laplace distribution has a sharp spike at zero вАФ many coefficients land exactly there.

**ElasticNet вЖФ mixture** of Gaussian and Laplace priors.

**The gotcha (for senior roles):** MAP only gives the posterior *mode* вАФ not the full posterior. The genuine Bayesian Lasso (Park & Casella 2008) uses a scale mixture of normals for the full posterior. Google Research interviewers may ask about this distinction.

</InterviewQuestion>

<InterviewQuestion
  number={10}
  difficulty="Conceptual"
  company={["Amazon", "Meta", "Google"]}
  question="How do you handle class imbalance in logistic regression while preserving calibration?"
>

Confirmed at Amazon MLE (Prepfully), Meta, and Google. Almost always a follow-up after any fraud, churn, or CTR modelling question.

**The core tension:** most fixes for imbalance hurt calibration. A strong answer addresses both.

**Fix 1 вАФ Class weighting:**
Set `class_weight='balanced'` in sklearn. The loss penalises minority-class errors proportionally to the class ratio. Preserves calibration reasonably well.

**Fix 2 вАФ Undersampling with calibration correction:**
Undersample the majority class by factor ќ≤ during training. After training, correct predicted probabilities:

`p_true = p_model / (p_model + (1 вИТ p_model) / ќ≤)`

Without this correction, predicted probabilities are systematically inflated for the minority class. Meta uses this in production for down-sampled CTR training data.

**Fix 3 вАФ SMOTE:**
Generates synthetic minority-class examples. Effective for decision boundary quality but **distorts calibration** (per JAMIA 2022). Apply Platt scaling or isotonic regression post-hoc.

**Threshold tuning:**
The default 0.5 threshold is wrong under imbalance. Tune it on a held-out validation set by optimising your business metric вАФ Youden's J, F1, or a cost-sensitive threshold.

**Evaluation metrics:**
Accuracy is useless. Use PR-AUC, F1/F-beta, log-loss, and the Brier score. Meta additionally uses **Normalized Entropy (NE)** from their 2014 ads paper.

</InterviewQuestion>

<InterviewQuestion
  number={11}
  difficulty="Conceptual"
  company={["Amazon", "Meta", "Google"]}
  question="When does logistic regression fail? Walk through at least three distinct failure modes."
>

**1. Perfect or quasi-complete separation:**
If a feature perfectly separates the classes, MLE diverges вАФ weights grow to ¬±вИЮ and the optimizer never converges. Fix: Firth's penalized likelihood, L2 regularization, or dropping the separating feature.

**2. Non-linearly separable structure:**
The decision boundary `wбµАx + b = 0` is always a hyperplane. XOR is the textbook example вАФ no linear boundary can separate it. Fix: engineer polynomial/interaction features, switch to a kernel SVM, tree-based model, or neural network.

**3. Severe multicollinearity:**
Highly correlated features produce unstable, high-variance coefficient estimates вАФ signs can flip unpredictably. Fix: Ridge regularization or drop one of the correlated features.

**4. Rare events with small n:**
With few positive examples, MLE estimates are biased away from zero (Hauck-Donner effect). Fix: Firth's correction or a Bayesian approach with an informative prior.

**5. Correlated observations:**
Logistic regression assumes i.i.d. observations. Clustered or time-series data violates this, inflating false-positive rates. Fix: mixed-effects logistic regression (GLMM) or GEE.

</InterviewQuestion>

<InterviewQuestion
  number={12}
  difficulty="System Design"
  company={["Meta", "Google"]}
  question="Design a CTR prediction system for an ads or search ranking platform. Why would you use logistic regression over GBDT or a deep model? How would you calibrate it?"
>

The archetypal Meta MLE system design question, and a confirmed Google MLE system design variant.

**Why logistic regression as the baseline:**

- Scores in **microseconds** вАФ critical at billion-query/day scale
- **Calibrated probabilities** by design вАФ directly optimising BCE gives calibrated outputs, essential for bid pricing in ads auctions
- **Interpretable** вАФ feature weights are auditable for policy and legal review
- **Works well in high-dimensional sparse feature spaces** вАФ text, categorical, ad IDs hashed via feature hashing

Meta's 2014 production paper used GBDT to engineer features, then logistic regression to score them вАФ not GBDT alone.

**Feature design:**

- User features: historical CTR, device, location, time of day, session depth
- Item/ad features: position bias, historical CTR of the ad, ad category
- Cross features: user √Ч ad category affinity, query √Ч ad relevance score
- Trend features: change in engagement last 7 days vs prior 7 days вАФ delta features often outperform absolute values

**Training:**

- Loss: binary cross-entropy
- Down-sample negatives (CTR is 1вАУ5%) for compute efficiency
- Re-calibrate after downsampling: `p_true = p_model / (p_model + (1 вИТ p_model) / ќ≤)`
- L2 regularization to control high-cardinality feature overfitting

**Serving at scale:**

- Precompute user and item features offline; retrieve at serve time from a feature store (Redis / Memcached)
- Shadow-deploy new model versions; A/B test on a traffic slice
- Monitor for distribution shift weekly вАФ CTR patterns shift with seasonality and ad inventory changes

**When to move beyond LR:**

Move to GBDT when non-linear feature interactions dominate. Move to a deep model (DLRM, two-tower) when learned embeddings for high-cardinality entities outperform engineered features. In practice, **LR remains the production baseline at Meta and Amazon in 2026** for latency-critical serving paths.

</InterviewQuestion>
