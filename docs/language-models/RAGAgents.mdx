# Retrieval-Augmented Generation (RAG) Agents: Connecting LLMs with Knowledge

## Learning Objective

By the end of this module, you will understand how RAG (Retrieval-Augmented Generation) architectures augment LLMs with external, factual knowledge by retrieving context dynamically from vector databases, mitigating hallucinations and improving accuracy on knowledge-intensive tasks.

---

## Is Every AI Agent a RAG Agent?

### Short Answer: **No**

This is a common misconception! While RAG agents are a popular and powerful type of AI agent, they represent just **one architectural pattern** among many. Let's clarify:

### Types of AI Agents

```
┌─────────────────────────────────────────────────────┐
│                   AI AGENTS                         │
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐ │
│  │   Simple     │  │   Tool-Using │  │   RAG    │ │
│  │   Prompt     │  │   Agents     │  │  Agents  │ │
│  │   Agents     │  │   (ReAct)    │  │          │ │
│  └──────────────┘  └──────────────┘  └──────────┘ │
│                                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────┐ │
│  │   Planning   │  │  Multi-Agent │  │  Memory  │ │
│  │   Agents     │  │   Systems    │  │  Agents  │ │
│  └──────────────┘  └──────────────┘  └──────────┘ │
└─────────────────────────────────────────────────────┘
```

#### 1. **Simple Prompt-Based Agents**
- Use only the LLM with carefully crafted prompts
- No external tools or retrieval
- Example: A creative writing assistant that just generates text

#### 2. **Tool-Using Agents (ReAct, Function Calling)**
- Can call external APIs, calculators, search engines
- Make decisions about which tools to use
- Example: An agent that searches the web, does calculations, sends emails

#### 3. **RAG Agents** (This Module's Focus)
- Retrieve relevant documents from a knowledge base
- Ground responses in retrieved context
- Example: A chatbot that answers questions about your company's documentation

#### 4. **Planning Agents**
- Break down complex tasks into sub-tasks
- Execute multi-step plans
- Example: Travel planning agents, research assistants

#### 5. **Multi-Agent Systems**
- Multiple agents collaborate or compete
- Each agent may have different roles
- Example: AutoGPT, BabyAGI, MetaGPT

#### 6. **Memory-Enabled Agents**
- Maintain long-term conversation history
- Learn from past interactions
- May or may not use retrieval

### When to Use RAG vs. Other Agent Types

| Scenario | Best Agent Type |
|----------|----------------|
| Answering questions from documents | **RAG Agent** |
| Booking flights and hotels | **Tool-Using Agent** |
| Writing creative stories | **Simple Prompt Agent** |
| Researching and writing a report | **Planning Agent + RAG** |
| Customer support with knowledge base | **RAG Agent + Tools** |
| Debugging code with web search | **Tool-Using Agent** |

**Key Takeaway**: RAG is a specific technique for **knowledge retrieval**. An agent might use RAG as one of several capabilities, or it might not need RAG at all!

## 1. The Problem: LLMs Forget Facts

### Static Knowledge Cutoff

Large Language Models are trained on massive datasets, but their knowledge is **frozen in time**:

- **GPT-4 (original)**: Training data cutoff in September 2021
- **Llama-2**: Training data cutoff in July 2023
- **Claude 3**: Training data cutoff varies by version

**What this means**:
```python
# Asking GPT-4 (trained on 2021 data) in 2024:
query = "Who won the 2024 Olympics?"
response = llm(query)
# Response: "I don't have information about events after 2021..."
```

### Hallucination Problems

When LLMs don't know something, they often **fabricate plausible-sounding but incorrect information**:

**Example of Hallucination**:
```
User: "What is the capital of the fictional country Elbonia?"

LLM (hallucinating): "The capital of Elbonia is Mudville, 
a bustling metropolis located along the Elbonian River..."

Reality: Elbonia is a fictional country from Dilbert comics!
```

### Why This Happens

1. **Training objective**: LLMs are trained to predict the next token, not to say "I don't know"
2. **Pattern matching**: They confuse similar patterns from training data
3. **Overconfidence**: No built-in uncertainty mechanism
4. **No grounding**: No connection to factual databases or real-world verification

### Real-World Consequences

- **Medical chatbots** giving incorrect drug interactions
- **Legal assistants** citing non-existent case law
- **Customer support** providing outdated product information
- **Financial advisors** missing recent market changes

### The Need for External Grounding

We need a way to:
1. ✅ Connect LLMs to **up-to-date information**
2. ✅ Ground responses in **verifiable sources**
3. ✅ Reduce hallucinations with **factual context**
4. ✅ Enable **dynamic knowledge updates** without retraining

**This is where RAG comes in.**

## 2. The Core Idea of RAG

### The RAG Paradigm

RAG (Retrieval-Augmented Generation) combines two complementary components:

1. **Retriever**: Searches and fetches relevant information from an external knowledge base
2. **Generator**: Uses the retrieved context to produce grounded, factual responses

### The Basic Formula

$$
\text{Answer} = \text{LLM}(\text{Query} + \text{Retrieved Context})
$$

Or more explicitly:

$$
\text{Answer} = \text{Generator}(\text{Prompt}(\text{Query}, \text{Retrieve}(\text{Query}, \text{Knowledge Base})))
$$

### How It Works (High-Level)

```
┌─────────────────────────────────────────────────────┐
│ Step 1: User Query                                  │
│ "What are the side effects of Drug X?"             │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│ Step 2: Retriever                                   │
│ - Convert query to embedding vector                 │
│ - Search knowledge base (medical docs)              │
│ - Return top-k most relevant chunks                 │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│ Step 3: Retrieved Context                           │
│ "Drug X may cause: nausea (10%), headache (5%)...   │
│  Clinical trials showed... FDA approved for..."     │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│ Step 4: Augmented Prompt                            │
│ "Given this context: [retrieved text]               │
│  Answer: What are the side effects of Drug X?"     │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│ Step 5: Generator (LLM)                             │
│ Produces answer grounded in retrieved context       │
└────────────┬────────────────────────────────────────┘
             │
             ▼
┌─────────────────────────────────────────────────────┐
│ Step 6: Final Answer                                │
│ "Based on clinical data, Drug X commonly causes     │
│  nausea (10% of patients) and headache (5%)..."    │
└─────────────────────────────────────────────────────┘
```

### Why This is Powerful

**Before RAG**:
- ❌ LLM guesses based on training data (possibly outdated or wrong)
- ❌ High risk of hallucination
- ❌ No source attribution

**With RAG**:
- ✅ LLM answers based on **actual retrieved documents**
- ✅ Reduced hallucinations (grounded in facts)
- ✅ Can cite sources (transparency)
- ✅ Knowledge updates don't require retraining

## 3. System Design Overview

### End-to-End RAG Workflow

Let's walk through a concrete example:

**User Question**: *"Who is the current CEO of OpenAI?"*

#### Step 1: Query Processing

```python
user_query = "Who is the current CEO of OpenAI?"
```

#### Step 2: Query Embedding

Convert the natural language query into a dense vector representation:

```python
# Using an embedding model (e.g., sentence-transformers)
query_embedding = embedding_model.encode(user_query)
# Result: [0.123, -0.456, 0.789, ...] (384 or 768 dimensions)
```

#### Step 3: Similarity Search in Vector Database

Search the vector database for documents with similar embeddings:

```python
# Vector DB (FAISS, Chroma, Pinecone, Weaviate)
results = vector_db.similarity_search(
    query_embedding,
    k=5  # Retrieve top 5 most similar documents
)
```

**Retrieved Documents** (example):
```
[
  {
    "text": "On November 17, 2023, OpenAI announced that Sam Altman 
             would step down as CEO...",
    "source": "openai-news-nov-2023.pdf",
    "score": 0.92
  },
  {
    "text": "Sam Altman returned as CEO of OpenAI on November 22, 2023, 
             following a board restructuring...",
    "source": "techcrunch-openai-update.html",
    "score": 0.89
  },
  ...
]
```

#### Step 4: Context Construction

Combine retrieved documents into a context string:

```python
context = "\n\n".join([doc["text"] for doc in results])
```

#### Step 5: Prompt Augmentation

Insert the retrieved context into the LLM prompt:

```python
prompt = f"""
You are a helpful assistant. Answer the question based ONLY on the 
following context. If the answer is not in the context, say 
"I don't have enough information to answer."

Context:
{context}

Question: {user_query}

Answer:
"""
```

#### Step 6: Generation

Pass the augmented prompt to the LLM:

```python
response = llm.generate(prompt)
# Output: "Based on the latest information, Sam Altman returned 
#          as CEO of OpenAI on November 22, 2023, after a brief 
#          period where he had stepped down..."
```

### Architecture Diagram

```
┌───────────────────────────────────────────────────────┐
│                    RAG SYSTEM                         │
│                                                       │
│  ┌─────────────┐         ┌──────────────────────┐   │
│  │   User      │────────▶│  Query Processor     │   │
│  │   Query     │         │  (Embedding Model)   │   │
│  └─────────────┘         └──────────┬───────────┘   │
│                                      │               │
│                                      ▼               │
│                          ┌──────────────────────┐   │
│                          │   Vector Database    │   │
│                          │  (FAISS/Chroma/      │   │
│                          │   Pinecone/Weaviate) │   │
│                          └──────────┬───────────┘   │
│                                      │               │
│                                      ▼               │
│                          ┌──────────────────────┐   │
│                          │  Top-K Documents     │   │
│                          │  (Retrieved Context) │   │
│                          └──────────┬───────────┘   │
│                                      │               │
│                                      ▼               │
│  ┌─────────────┐         ┌──────────────────────┐   │
│  │  Generated  │◀────────│   LLM Generator      │   │
│  │  Answer     │         │   (GPT-4/Claude/     │   │
│  └─────────────┘         │    Llama)            │   │
│                          └──────────────────────┘   │
└───────────────────────────────────────────────────────┘
```

### Key Parameters

- **k (top-k)**: Number of documents to retrieve (typically 3-10)
- **Chunk size**: Length of document chunks (typically 256-1024 tokens)
- **Chunk overlap**: Overlap between chunks (typically 10-20% of chunk size)
- **Similarity threshold**: Minimum similarity score to include a document

## 4. Components of a RAG Pipeline

### Component 1: Document Loader

Ingests documents from various sources:

```python
from langchain.document_loaders import (
    PyPDFLoader,
    TextLoader,
    UnstructuredMarkdownLoader,
    WebBaseLoader
)

# Load PDFs
pdf_loader = PyPDFLoader("company-handbook.pdf")
docs_pdf = pdf_loader.load()

# Load from web
web_loader = WebBaseLoader("https://docs.company.com")
docs_web = web_loader.load()

# Load markdown
md_loader = UnstructuredMarkdownLoader("README.md")
docs_md = md_loader.load()
```

### Component 2: Text Splitter

Breaks long documents into smaller, manageable chunks:

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,        # Characters per chunk
    chunk_overlap=200,      # Overlap between chunks
    separators=["\n\n", "\n", " ", ""]  # Split hierarchy
)

chunks = text_splitter.split_documents(docs_pdf)
# Result: [Document(text="...", metadata={...}), ...]
```

**Why Chunking Matters**:
- LLMs have **context window limits** (4k-128k tokens)
- Smaller chunks = more **precise retrieval**
- Overlap ensures **no information loss** at boundaries

### Component 3: Embeddings Model

Converts text chunks into dense vector representations:

```python
from langchain.embeddings import (
    OpenAIEmbeddings,      # text-embedding-ada-002
    HuggingFaceEmbeddings, # all-MiniLM-L6-v2, e5-large
    CohereEmbeddings       # embed-english-v3.0
)

# Using OpenAI
embeddings = OpenAIEmbeddings(model="text-embedding-ada-002")

# Using open-source (Sentence Transformers)
embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-mpnet-base-v2"
)
```

**Popular Embedding Models**:

| Model | Dimensions | Use Case |
|-------|-----------|----------|
| OpenAI ada-002 | 1536 | General purpose, high quality |
| all-MiniLM-L6-v2 | 384 | Fast, lightweight, open-source |
| all-mpnet-base-v2 | 768 | Better quality, open-source |
| e5-large-v2 | 1024 | State-of-art open-source |
| Cohere embed-v3 | 1024 | Multilingual, commercial |

### Component 4: Vector Store

Stores and retrieves embeddings efficiently:

```python
from langchain.vectorstores import (
    Chroma,      # Local, persistent
    FAISS,       # Facebook AI, fast
    Pinecone,    # Cloud, scalable
    Weaviate,    # Open-source, production
    Qdrant       # Rust-based, high-performance
)

# Example: Chroma (local development)
from langchain.vectorstores import Chroma

vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)
vectorstore.persist()

# Example: Pinecone (cloud production)
import pinecone
from langchain.vectorstores import Pinecone

pinecone.init(api_key="YOUR_API_KEY", environment="us-west1-gcp")
vectorstore = Pinecone.from_documents(
    documents=chunks,
    embedding=embeddings,
    index_name="my-rag-index"
)
```

**Vector Store Comparison**:

| Store | Type | Best For |
|-------|------|----------|
| **FAISS** | In-memory | Fast local experiments |
| **Chroma** | Persistent local | Development, small datasets |
| **Pinecone** | Cloud | Production, scalability |
| **Weaviate** | Self-hosted/Cloud | Enterprise, flexibility |
| **Qdrant** | Self-hosted/Cloud | High performance, filtering |

### Component 5: Retriever

Defines how documents are fetched:

```python
# Basic similarity search
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)

# Maximum Marginal Relevance (MMR) - reduces redundancy
retriever = vectorstore.as_retriever(
    search_type="mmr",
    search_kwargs={"k": 5, "fetch_k": 20}
)

# Similarity with score threshold
retriever = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"score_threshold": 0.7, "k": 5}
)
```

**Retrieval Strategies**:
- **Similarity Search**: Pure cosine similarity
- **MMR (Maximum Marginal Relevance)**: Balances relevance and diversity
- **Score Threshold**: Only retrieve above a confidence level

### Component 6: Prompt Template

Structures how retrieved context is presented to the LLM:

```python
from langchain.prompts import PromptTemplate

template = """
You are an expert assistant. Use the following pieces of context 
to answer the question at the end. If you don't know the answer, 
just say you don't know. Don't make up an answer.

Context:
{context}

Question: {question}

Helpful Answer:"""

prompt = PromptTemplate(
    template=template,
    input_variables=["context", "question"]
)
```

**Prompt Engineering Tips**:
- ✅ Explicitly tell the model to use the context
- ✅ Instruct it NOT to hallucinate
- ✅ Ask for citations when needed
- ✅ Specify the desired output format

## 5. Example Implementation

### Full RAG System with LangChain

```python
from langchain.chains import RetrievalQA
from langchain.chat_models import ChatOpenAI
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

# Step 1: Load documents
loader = PyPDFLoader("company-docs.pdf")
documents = loader.load()

# Step 2: Split into chunks
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,
    chunk_overlap=200
)
chunks = text_splitter.split_documents(documents)

# Step 3: Create embeddings and vector store
embeddings = OpenAIEmbeddings()
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./data"
)

# Step 4: Create retriever
retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5}
)

# Step 5: Create LLM
llm = ChatOpenAI(model="gpt-4", temperature=0)

# Step 6: Create RAG chain
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # "stuff", "map_reduce", "refine"
    retriever=retriever,
    return_source_documents=True
)

# Step 7: Query the system
response = qa_chain({
    "query": "What is the latest paper on diffusion models?"
})

print("Answer:", response["result"])
print("Sources:", response["source_documents"])
```

### Full RAG System with LlamaIndex

```python
from llama_index import (
    VectorStoreIndex,
    SimpleDirectoryReader,
    ServiceContext,
    StorageContext,
    load_index_from_storage
)
from llama_index.embeddings import OpenAIEmbedding
from llama_index.llms import OpenAI

# Step 1: Load documents
documents = SimpleDirectoryReader("./data").load_data()

# Step 2: Configure service context
service_context = ServiceContext.from_defaults(
    llm=OpenAI(model="gpt-4"),
    embed_model=OpenAIEmbedding()
)

# Step 3: Build index
index = VectorStoreIndex.from_documents(
    documents,
    service_context=service_context
)

# Step 4: Create query engine
query_engine = index.as_query_engine(
    similarity_top_k=5,
    response_mode="compact"
)

# Step 5: Query
response = query_engine.query(
    "What is the latest paper on diffusion models?"
)

print("Answer:", response.response)
print("Sources:", response.source_nodes)
```

### Advanced: Custom RAG with Direct Vector DB Access

```python
import openai
from sentence_transformers import SentenceTransformer
import chromadb

# Initialize
embedding_model = SentenceTransformer('all-mpnet-base-v2')
client = chromadb.Client()
collection = client.create_collection("my_docs")

# Add documents
texts = [
    "Diffusion models are a class of generative models...",
    "Stable Diffusion was released in 2022...",
    # ... more documents
]
embeddings = embedding_model.encode(texts)
collection.add(
    embeddings=embeddings.tolist(),
    documents=texts,
    ids=[f"doc_{i}" for i in range(len(texts))]
)

# Query
def rag_query(question):
    # 1. Embed query
    query_embedding = embedding_model.encode([question])[0]
    
    # 2. Retrieve
    results = collection.query(
        query_embeddings=[query_embedding.tolist()],
        n_results=5
    )
    
    # 3. Build context
    context = "\n\n".join(results["documents"][0])
    
    # 4. Generate
    prompt = f"""
    Context: {context}
    
    Question: {question}
    
    Answer based on the context above:
    """
    
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": prompt}]
    )
    
    return response.choices[0].message.content

# Use it
answer = rag_query("What are diffusion models?")
print(answer)
```

## 6. Advantages of RAG

### 1. Grounded and Factual Answers

**Without RAG**:
```
User: "What is our company's return policy?"
LLM: "Most companies allow returns within 30 days..."  ❌ Generic
```

**With RAG**:
```
User: "What is our company's return policy?"
LLM: "According to our policy document, we accept returns 
      within 45 days with original receipt..."  ✅ Specific
```

### 2. Dynamic Knowledge Updates

- **No retraining needed**: Just update the knowledge base
- **Real-time information**: Add new documents instantly
- **Cost-effective**: Avoid expensive model fine-tuning

**Example**: Company launches new product
- ❌ **Without RAG**: Wait weeks/months for model retraining
- ✅ **With RAG**: Add product docs to vector store → works immediately

### 3. Source Attribution and Transparency

```python
response = qa_chain({"query": "What causes climate change?"})

print("Answer:", response["result"])
print("Sources:")
for doc in response["source_documents"]:
    print(f"  - {doc.metadata['source']}, page {doc.metadata['page']}")
```

**Output**:
```
Answer: Climate change is primarily caused by greenhouse gas emissions...
Sources:
  - ipcc-report-2023.pdf, page 15
  - climate-science.pdf, page 203
```

This enables:
- **Verification**: Users can check sources
- **Trust**: Transparency builds confidence
- **Compliance**: Critical for regulated industries (legal, medical)

### 4. Reduced Hallucinations

| Metric | Without RAG | With RAG |
|--------|------------|----------|
| Factual accuracy | 60-70% | 85-95% |
| Hallucination rate | 30-40% | 5-15% |
| Source-grounded responses | 0% | 100% |

### 5. Domain Specialization Without Fine-Tuning

- **Cheaper**: No GPU costs for training
- **Faster**: Deploy in hours, not weeks
- **Flexible**: Easy to add/remove/update knowledge
- **Multiple domains**: One model, many knowledge bases

### 6. Privacy and Control

- **On-premise deployment**: Keep sensitive data in-house
- **Access control**: Fine-grained permissions on documents
- **Data sovereignty**: Comply with regional regulations

### 7. Scalability

- Vector databases scale horizontally
- Can handle millions of documents
- Incremental updates (add documents without rebuilding)

## 7. Limitations

### 1. Dependent on Retrieval Quality

**Garbage In, Garbage Out**:
- If retrieval fails to find relevant docs → poor answers
- If retrieved docs are incorrect → model amplifies errors

**Example**:
```
User: "What is the boiling point of water?"

Retrieved (wrong): "Water boils at 90°C in Denver..."
LLM: "Water boils at 90°C."  ❌ Incorrect (context altitude)
```

**Mitigation**:
- Use high-quality embeddings models
- Implement hybrid search (vector + keyword)
- Add reranking stage (e.g., cross-encoder models)

### 2. Slower Inference Due to I/O

**Latency Breakdown**:
```
Total: 2-5 seconds
├─ Embedding query: 50-100ms
├─ Vector search: 100-500ms
├─ LLM generation: 1-3 seconds
└─ Overhead: 100-200ms
```

Compare to **direct LLM** (no RAG): 1-2 seconds

**Mitigation**:
- Cache frequent queries
- Use faster vector databases (FAISS, Qdrant)
- Implement parallel retrieval
- Optimize chunk sizes

### 3. No Permanent Parameter Adaptation

- Model doesn't "learn" from retrieved information
- Knowledge is ephemeral (only in prompt context)
- Can't internalize patterns across many queries

**Contrast with Fine-Tuning**:
- Fine-tuning updates weights → permanent learning
- RAG retrieves context → temporary grounding

**When Each is Better**:
- **Use RAG**: Dynamic facts, frequent updates, source attribution
- **Use Fine-Tuning**: Learn patterns/style, improve general capabilities

### 4. Context Window Limitations

Even with long context models:
- GPT-4: 8k-128k tokens
- Claude 3: 200k tokens
- Llama-3: 8k-128k tokens

You can't retrieve unlimited documents!

**Trade-offs**:
- Retrieve few chunks (k=3-5): Fast but might miss info
- Retrieve many chunks (k=20-50): Better coverage but slower, more noise

### 5. Chunking Challenges

**Problem**: Information spans multiple chunks
```
Chunk 1: "The CEO of Company X is John..."
Chunk 2: "...Smith, who joined in 2020."

Query: "Who is CEO of Company X?"
Retrieved: Only Chunk 1 → Answer: "John" (incomplete)
```

**Mitigation**:
- Use overlap between chunks
- Implement parent-document retrieval
- Use smaller, more granular chunks

### 6. Cost Considerations

- **Embedding API costs**: $0.0001-0.0004 per 1K tokens
- **Vector DB hosting**: $0.10-1.00 per GB/month
- **LLM API costs**: $0.001-0.06 per 1K tokens

For **1M queries/month** with average 3k tokens:
- Embeddings: ~$120-360
- LLM: ~$3,000-180,000 (depending on model)
- Vector DB: ~$50-500

### 7. Retrieval Failures

**When retrieval doesn't work well**:
- **Multi-hop reasoning**: "Who is the spouse of the CEO of Company X?"
  - Requires 2 retrievals: find CEO → find spouse
- **Aggregation queries**: "What is the average salary?"
  - Retrieval returns examples, not aggregated data
- **Comparison queries**: "Compare product A vs B"
  - Need to retrieve info for both, then synthesize

**Solutions**:
- Use multi-step RAG (ReAct-style agents)
- Combine RAG with tool use (calculators, SQL)
- Implement query decomposition

## 8. Where It's Used

### Enterprise Knowledge Management

**Use Case: Internal Company Q&A**
```
Scenario: 10,000 employee company with:
- 5,000 internal documents
- HR policies, engineering docs, product specs
- Multiple departments, constant updates

Solution: RAG chatbot
- Employees ask questions in natural language
- Instant answers with source citations
- No need to search through document repositories
- Automatic updates when docs change
```

**Benefits**:
- ✅ Reduced time searching for information
- ✅ Democratized access to knowledge
- ✅ Onboarding new employees faster

### Customer Support and Chatbots

**Use Case: Technical Support Bot**
```
Knowledge Base:
- Product manuals
- Troubleshooting guides
```

### The RAG Workflow Pipeline

```
1. INDEXING (One-time / Periodic)
   ↓
   Documents → Split → Embed → Store in Vector DB
   
2. RETRIEVAL (Per Query)
   ↓
   Query → Embed → Similarity Search → Top-K Docs
   
3. GENERATION (Per Query)
   ↓
   Query + Retrieved Docs → Prompt → LLM → Answer
```

## 10. Advanced RAG Techniques

### Multi-Query RAG

Generate multiple query variations for better retrieval:

```python
from langchain.retrievers import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(),
    llm=ChatOpenAI(temperature=0)
)

# Single query → Multiple query variations → Better retrieval
```

### Hypothetical Document Embeddings (HyDE)

Generate a hypothetical answer, then search for documents similar to it:

```
Query: "What is quantum entanglement?"
  ↓
LLM: Generate hypothetical answer (may be imperfect)
  ↓
Embed hypothetical answer
  ↓
Search for docs similar to hypothetical answer
  ↓
Use actual docs to generate real answer
```

### Parent Document Retrieval

Retrieve small chunks, but use larger parent documents for context:

```python
from langchain.retrievers import ParentDocumentRetriever

retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,
    docstore=docstore,
    child_splitter=small_splitter,    # 200 tokens
    parent_splitter=large_splitter    # 2000 tokens
)
```

### Self-Query Retrieval

LLM extracts metadata filters from query:

```
Query: "Show me Python tutorials from 2023"
  ↓
LLM extracts: {language: "Python", year: 2023}
  ↓
Retrieval with filters: vectorstore.search(filters={...})
```

## Summary

### Key Takeaways

1. **RAG is not a type of agent, but a technique** that agents can use for knowledge retrieval
2. **RAG = Retriever + Generator**: Combines external knowledge with LLM generation
3. **Solves hallucination problem**: Grounds responses in factual, retrieved documents
4. **No retraining needed**: Update knowledge by updating the vector database
5. **Trade-offs**: Better accuracy vs. slower inference and retrieval dependency
6. **Wide applicability**: Customer support, legal, finance, healthcare, research

### When to Use RAG

✅ **Use RAG when**:
- You need factual, up-to-date information
- Knowledge changes frequently
- Source attribution is important
- You have a specific document corpus
- Hallucinations are unacceptable

❌ **Don't use RAG when**:
- Pure creative generation (fiction, poetry)
- Mathematical reasoning (use tools instead)
- General knowledge (LLM alone is sufficient)
- Real-time data needed (use APIs/tools instead)

### RAG vs. Other Approaches

| Approach | Best For | Limitations |
|----------|----------|-------------|
| **RAG** | Factual Q&A, document search | Retrieval quality dependency |
| **Fine-Tuning** | Task-specific behavior, style | Static knowledge, expensive |
| **Prompt Engineering** | General tasks, quick iteration | Limited context, no external data |
| **Tool Use** | APIs, calculations, actions | Requires predefined tools |

### The Future of RAG

- **Hybrid search**: Combining vector + keyword + graph search
- **Multi-modal RAG**: Retrieving images, videos, audio
- **Agentic RAG**: RAG + planning + tool use
- **Collaborative RAG**: Multiple retrievers, cross-document reasoning
- **Real-time RAG**: Streaming retrieval and generation
