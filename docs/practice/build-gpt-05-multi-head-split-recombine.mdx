  solution={`
import numpy as np

def split_heads(x, n_head):
    B, T, C = x.shape
    assert C % n_head == 0, "n_head must divide C"
    D = C // n_head
    # (B, T, C) -> (B, T, H, D) -> (B, H, T, D)
    return x.reshape(B, T, n_head, D).transpose(0, 2, 1, 3)


def combine_heads(x):
    B, H, T, D = x.shape
    # (B, H, T, D) -> (B, T, H, D) -> (B, T, H * D)
    return x.transpose(0, 2, 1, 3).reshape(B, T, H * D)
`}

>

The reshape pair every transformer uses to run `n_head` parallel attention computations on the same `(B, T, C)` tensor. The trick is the order: **reshape, then transpose** for split; **transpose, then reshape** for combine.

The shapes:

`split:    (B, T, C) → (B, T, H, D) → (B, H, T, D)`

`combine:  (B, H, T, D) → (B, T, H, D) → (B, T, H * D)`

Why this order matters: a reshape just relabels the bytes — `(B, T, C)` becomes `(B, T, H, D)` *without moving any data*. The transpose then physically rearranges so each head's data is contiguous along its own row. If you transpose first then reshape, the bytes get interpreted in the wrong order — the diagnostic "split then combine == identity" test catches this immediately.

Both functions ship as one line each (assertion plus reshape-transpose), but getting the axis order right is the whole point. After this problem, `build-gpt-06-multi-head-attention` calls these to wrap the single-head `attention` from problem 03.
