import numpy as np

def batch_norm_forward(x, gamma, beta, running_mean, running_var,
                       momentum, eps, training):
    if training:
        mu = x.mean(axis=0)
        var = x.var(axis=0)        # biased (1/N), matches normalisation
        x_hat = (x - mu) / np.sqrt(var + eps)
        # Update running stats in place.
        running_mean *= (1 - momentum); running_mean += momentum * mu
        running_var  *= (1 - momentum); running_var  += momentum * var
    else:
        x_hat = (x - running_mean) / np.sqrt(running_var + eps)
    return gamma * x_hat + beta
`}

>

BatchNorm — the original normalisation layer, introduced by [Ioffe & Szegedy (2015)](https://arxiv.org/abs/1502.03167). Two distinct behaviours that the harness checks separately:

**Train mode:** compute batch statistics across `axis=0`, normalise the input, **and update running statistics** for use at eval time. The running stats are exponentially-moving averages with weight `momentum` on the new batch stats and `(1 - momentum)` on the previous running value.

**Eval mode:** use the saved running statistics — don't peek at the eval batch. This is what makes inference deterministic and stable: a single example produces the same output regardless of what other examples are in its (often nonexistent) batch.

The two-mode design is also why BatchNorm fails in autoregressive language modelling: at inference you generate one token at a time, so "the eval batch" is a batch of size 1 where stats are degenerate. That's why GPT-2 uses LayerNorm (which has no batch axis) instead.

The diagnostic the harness probes: in train mode with `gamma=1, beta=0`, each feature's output has `mean ≈ 0, std ≈ 1` along the batch axis. In eval mode, the running stats *don't* change after the call.
