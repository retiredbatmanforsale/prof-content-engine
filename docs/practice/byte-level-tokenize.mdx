def byte_encode(text):
    return list(text.encode("utf-8"))


def byte_decode(ids):
    return bytes(ids).decode("utf-8", errors="replace")

The simplest tokenizer that works for any unicode text: encode the string as UTF-8, then treat each byte as its own token. Vocab size is exactly 256.

This is the foundation of GPT-2's byte-level BPE — the BPE step is *built on top of* this byte tokenizer, so even tokens that don't appear in the BPE vocabulary still survive the encode/decode round-trip via their raw bytes.

Two operations:

1. **`byte_encode(text)`** — UTF-8 encode the string, take the resulting bytes as a list of ints.
2. **`byte_decode(ids)`** — inverse: bytes back to a UTF-8 string.

Three things the harness checks:

1. **Round-trip on ASCII** — the easy case. `"Hi" → [72, 105] → "Hi"`.
2. **Round-trip on emoji and CJK** — the hard case. A single grinning face emoji (`U+1F600`) takes **four bytes** in UTF-8: `[0xF0, 0x9F, 0x98, 0x80]`. The encoder must produce those four bytes, and the decoder must reassemble them correctly.
3. **Decode robustness** — pass `errors='replace'` so a truncated byte sequence (which can happen with sliding-window LM inference) doesn't raise.

The trade-off vs subword BPE: byte-level tokens are universal but verbose — every character costs 1-4 tokens. BPE compresses common byte sequences into single tokens (GPT-2's vocab compresses ~50K English byte-pairs into ~50K vocab entries, so English text averages ~3.5 chars/token).
